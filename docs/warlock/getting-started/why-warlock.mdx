# Why Warlock?

Choosing a framework is a big decision. Let's talk about what makes Warlock.js different—and when it might (or might not) be the right fit for you.

## The Philosophy

Warlock.js is **opinionated for productivity**. We believe that:

- **Convention over configuration** gets you shipping faster
- **Integrated tooling** beats stitching together 20 packages
- **Developer experience** directly impacts product quality
- **Sensible defaults** should "just work" for 80% of use cases

If you prefer total flexibility and building everything from scratch, frameworks like Express might be a better fit. But if you want to focus on your product instead of infrastructure, keep reading.

## What Makes Warlock.js Different?

### 1. Batteries Included

Most frameworks give you routing and... that's it. Warlock.js ships with:

- **Database ORM** (MongoDB + PostgreSQL)
- **Authentication** (JWT, roles, permissions)
- **File Storage** (S3, local, with image processing)
- **Email System** (with React templates)
- **Caching** (Redis, Memory, File, and more)
- **Validation** (type-safe schemas)
- **Event Bus** (for decoupled architecture)

You don't need to research, compare, and integrate these yourself. They're already wired up.

### 2. Development Experience That Doesn't Suck

Warlock's dev server is **fast**:

- **Hot Module Reload** — Change code, see it instantly (no server restart)
- **Health Checks** — Know exactly what's broken and where
- **Type Generation** — Auto-generate types for your API
- **Smart File Watching** — Only rebuilds what changed

No `nodemon`. No waiting. Just code and see results.

### 3. Module-Based Architecture

Organize your code by **feature**, not by layer:

```
src/app/
├── users/
│   ├── controllers/
│   ├── models/
│   ├── routes.ts
│   └── events/
├── posts/
└── orders/
```

Each module is self-contained. Want to remove a feature? Delete the folder. Want to share code? Move it to `utils/`. Simple.

### 4. Type Safety Without the Ceremony

Validation that feels natural:

```typescript
import { v } from "@warlock.js/seal";

createPostController.validation = {
  schema: v.object({
    title: v.string().required().min(3),
    content: v.string().required(),
    tags: v.array(v.string()).optional(),
  }),
};
```

No decorators on every property. No separate DTO classes. Just clean, readable validation.

### 5. Production-Ready Out of the Box

Warlock.js includes features most frameworks make you build yourself:

- **Multi-Tenancy** — Built-in support for SaaS apps
- **Rate Limiting** — Protect your APIs from abuse
- **CORS** — Configured and ready
- **Error Handling** — Consistent error responses
- **Logging** — Structured logs with multiple transports
- **Health Checks** — Monitor your app's status

## For Teams Who Want...

Warlock.js is perfect if you:

- **Want to ship fast** — Spend time on features, not infrastructure
- **Value conventions** — Prefer "the Warlock.js way" over endless options
- **Need integrated tooling** — Don't want to maintain 20+ dependencies
- **Care about DX** — Developer happiness leads to better products
- **Build SaaS apps** — Multi-tenancy and auth are first-class citizens

## When Warlock.js Might Not Be Right

We're honest about trade-offs. Warlock.js might not be ideal if you:

- **Need maximum flexibility** — Warlock.js is opinionated by design
- **Want to use specific tools** — We have preferred solutions (e.g., Cascade ORM)
- **Have a tiny microservice** — Warlock's power shines in full-featured APIs
- **Prefer class-based everything** — We use functional controllers (though classes are supported)

## The Bottom Line

**Express** gives you freedom but requires you to build everything.  
**NestJS** gives you structure but can feel heavy and complex.  
**Warlock** gives you a complete, integrated toolkit with great DX.

If you want to build production-ready APIs without the configuration overhead, Warlock.js is built for you.
