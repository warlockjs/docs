# Project Structure

Knowing where to put your code is half the battle. Let's walk through a Warlock.js project's structure and explain what each directory does.

## Root Structure

After running `create-warlock`, you'll have this structure:

```
my-app/
├── src/                  # Source code
│   ├── app/              # Application modules
│   └── config/           # Configuration files
├── storage/              # File uploads, logs
├── .warlock/             # Dev server cache (auto-generated)
├── .env                  # Environment variables
├── .env.shared           # Shared env variables
├── .env.production       # Production env variables
├── warlock.config.ts     # Warlock.js configuration
├── package.json          # Dependencies
├── tsconfig.json         # TypeScript config
└── .gitignore            # Git ignore rules
```

Let's break down each part.

## `src/` Directory

All your application code lives here.

### `src/config/` — Configuration Files

Configuration files that transform environment variables into structured config:

```
src/config/
├── app.ts           # App name, locale, timezone
├── database.ts      # Database connection
├── http.ts          # HTTP server settings
├── auth.ts          # Authentication config
├── cache.ts         # Caching config
├── mail.ts          # Email config
├── uploads.ts       # File upload config
└── validation.ts    # Validation config
```

**Example:**

```typescript title="src/config/database.ts"
import { env } from "@warlock.js/core";

export default {
  driver: env("DB_DRIVER", "mongodb"),
  host: env("DB_HOST", "localhost"),
  port: env("DB_PORT", 27017),
  name: env("DB_NAME", "myapp"),
};
```

All files here are **auto-loaded** on startup. You don't need to import them manually.

### `src/app/` — Application Modules

This is where your features live. Each subdirectory is a **module**:

```
src/app/
├── users/           # User management
├── posts/           # Blog posts
├── orders/          # E-commerce orders
└── utils/           # Shared utilities
```

The `utils/` directory is special—it's for code shared across modules.

## Module Structure

Let's look at a typical module:

```
users/
├── controllers/          # Request handlers
│   ├── create-user.ts
│   ├── list-users.ts
│   └── get-user.ts
├── models/               # Database models
│   └── user.ts
├── repositories/         # Data access layer
│   └── users-repository.ts
├── output/               # Response transformers
│   └── user-output.ts
├── events/               # Event handlers
│   ├── user-created.ts
│   └── user-updated.ts
├── utils/                # Module utilities
│   ├── flags.ts          # Constants, enums
│   └── locales.ts        # Translations
├── routes.ts             # Route definitions
└── main.ts               # Module entry point (optional)
```

### Key Files

#### `routes.ts` — Route Definitions

Define all routes for this module:

```typescript title="src/app/users/routes.ts"
import { router } from "@warlock.js/core";
import { createUserController } from "./controllers/create-user";
import { listUsersController } from "./controllers/list-users";

router.post("/users", createUserController);
router.get("/users", listUsersController);
```

Routes are **auto-loaded** from all modules.

#### `main.ts` — Module Entry Point (Optional)

Use this for module-level initialization:

```typescript title="src/app/users/main.ts"
import { events } from "@warlock.js/core";

// Register event listeners
events.on("app.ready", () => {
  console.log("Users module initialized");
});
```

#### `utils/flags.ts` — Constants

Store module-specific constants:

```typescript title="src/app/users/utils/flags.ts"
export const USER_ROLES = {
  ADMIN: "admin",
  USER: "user",
  GUEST: "guest",
} as const;

export const USER_STATUS = {
  ACTIVE: "active",
  INACTIVE: "inactive",
  BANNED: "banned",
} as const;
```

#### `utils/locales.ts` — Translations

Define translations for this module:

```typescript title="src/app/users/utils/locales.ts"
import { groupedTranslations } from "@mongez/localization";

export default groupedTranslations({
  userCreated: {
    en: "User created successfully",
    ar: "تم إنشاء المستخدم بنجاح",
  },
  userNotFound: {
    en: "User not found",
    ar: "المستخدم غير موجود",
  },
  userInvalidEmail: {
    en: "Invalid email address",
    ar: "عنوان البريد الإلكتروني غير صالح",
  },
});
```

Locale files are **auto-loaded** from all modules.

## `storage/` Directory

Stores uploaded files, logs, and temporary data:

```
storage/
├── uploads/         # User-uploaded files
├── logs/            # Application logs
└── temp/            # Temporary files
```

This directory is **gitignored** by default.

## `.warlock/` Directory (v4)

The dev server cache. **Never edit this manually.**

```
.warlock/
├── cache/           # Transpiled TypeScript files
└── manifest.json    # File metadata
```

This directory is **auto-generated** and **gitignored**.

:::tip Why `.warlock/`?
Warlock's dev server transpiles your TypeScript files and caches them here. This makes HMR blazing fast because it only re-transpiles changed files.
:::

## Environment Files

### `.env` — Development Environment

Default environment variables for local development:

```env
# App
APP_NAME=myapp
HTTP_PORT=2030
HOST=localhost

# Database (MongoDB)
DB_DRIVER=mongodb
DB_HOST=localhost
DB_PORT=27017
DB_NAME=myapp
```

## `warlock.config.ts` — Warlock.js Configuration

Configure Warlock-specific settings:

```typescript title="warlock.config.ts"
import { defineConfig } from "@warlock.js/core";

export default defineConfig({
  build?: {
    /**
     * Output directory
     */
    outDirectory: "./dist",
    /**
     * Output file
     */
    outFile: "app.js",
    /**
     * Minify output
     */
    minify: true,
    /**
     * Generate sourcemap
     */
    sourcemap: true,
  };
});
```

Learn more in the [Dev Server](./dev-server-overview) section.

## Putting It All Together

Here's a complete example of a posts module:

```
src/app/posts/
├── controllers/
│   ├── create-post.ts       # POST /posts
│   ├── list-posts.ts        # GET /posts
│   └── get-post.ts          # GET /posts/:id
├── models/
│   └── post.ts              # Post database model
├── repositories/
│   └── posts-repository.ts  # Data access for posts
├── output/
│   └── post-output.ts       # Transform post for response
├── events/
│   └── post-created.ts      # Send email on post creation
├── utils/
│   ├── flags.ts             # POST_STATUS constants
│   └── locales.ts           # Post-related translations
└── routes.ts                # Define /posts routes
```

**Flow:**

1. User hits `GET /posts`
2. Warlock.js loads `routes.ts` and matches the route
3. Calls `listPostsController`
4. Controller uses `postsRepository` to fetch data
5. Repository queries `Post` model
6. Controller transforms data with `postOutput`
7. Returns JSON response

## Best Practices

### ✅ Do This

- **Keep modules focused** — One module = one feature
- **Use `utils/` for shared code** — Don't duplicate across modules
- **Follow naming conventions** — `create-user.ts`, not `CreateUser.ts`
- **Organize by feature** — Not by layer (controllers/, models/, etc.)

### ❌ Avoid This

- **Don't create giant modules** — Split large features into sub-modules
- **Don't put business logic in controllers** — Use services or repositories
- **Don't hardcode values** — Use `utils/flags.ts` for constants
- **Don't edit `.warlock/`** — It's auto-generated
