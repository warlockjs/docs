# Project Structure

Here's how a Warlock.js project is organized. Know this, and you'll always know where to put things.

## Root Structure

```
my-project/
├── src/
│   ├── app/              # Your application modules
│   └── config/           # Configuration files
├── storage/              # Files, logs (gitignored)
├── .warlock/             # Dev server cache (gitignored)
├── .env                  # Environment variables
├── warlock.config.ts     # Warlock configuration
├── tsconfig.json
├── eslint.config.js
└── package.json
```

## The `src/` Directory

### `src/config/`

All your configuration files. Loaded automatically on startup:

```
src/config/
├── app.ts          # App name, timezone, locale
├── database.ts     # Database connection
├── http.ts         # Server port, CORS
├── auth.ts         # JWT settings
├── mail.ts         # SMTP settings
├── cache.ts        # Redis/cache settings
├── storage.ts      # File storage drivers
├── log.ts          # Logging configuration
├── validation.ts   # Validation settings
└── tests.ts        # Test configuration
```

### `src/app/`

Your application code, organized by module:

```
src/app/
├── users/          # User module
├── posts/          # Posts module
├── auth/           # Authentication module
├── uploads/        # File uploads module
├── shared/         # Shared components
```

## Module Structure

Every module follows the same pattern. Here's a typical module:

```
src/app/users/
├── controllers/              # Request handlers
│   ├── list-users.controller.ts
│   ├── create-user.controller.ts
│   ├── get-user.controller.ts
│   ├── update-user.controller.ts
│   └── delete-user.controller.ts
├── events/                   # Events
├── models/                   # Database models
│   └── user/
│       ├── user.model.ts
│       └── migrations/       # Model migrations
├── repositories/             # Data access layer
│   └── users.repository.ts
├── services/                 # Business logic
│   └── user.service.ts
├── resources/                 # Resource transformers (For http response output)
│   └── user.resource.ts
├── events/                   # Event handlers
│   ├── index.ts              # Event exports
│   └── on-user-created.ts
├── commands/                 # CLI commands
│   └── create-admin.command.ts
├── seeds/                    # Database seeders
│   └── users.seeder.ts
├── utils/                    # Utilities
│   └── locales.ts            # Translations
│   └── flags.ts              # Static data
│   └── types.ts              # TypeScript types
├── routes.ts                 # Route definitions
└── main.ts                   # Module entry (optional)
```

### Special Files

These files are auto-loaded by the dev server:

| File            | Purpose               | When It Loads       |
| --------------- | --------------------- | ------------------- |
| `utils/locales` | Translations          | First to be loaded  |
| `events/`       | Event handlers        | Second to be loaded |
| `main.ts`       | Module initialization | Third to be loaded  |
| `routes.ts`     | Route definitions     | Fourth to be loaded |

### Loading Order

Warlock loads these files in a specific order:

```
1. src/config/*           # All configs first
2. src/app/main.ts            # App entry (if exists)
3. Per module:
   ├── utils/locales.ts            # Translations (if exists)
   ├── events/*           # Events files (if exists)
   ├── main.ts            # Module init (if exists)
   └── routes.ts          # Routes (if exists)
```

## The `storage/` Directory

Files that shouldn't be in version control:

```
storage/
├── uploads/        # Uploaded files
├── logs/           # Application logs
└── cache/          # File-based cache (if used)
```

Make sure this is in your `.gitignore`.

## The `.warlock/` Directory

Dev server internals. Completely managed by Warlock:

```
.warlock/
├── cache/          # Transpiled TypeScript files
└── manifest.json   # File metadata for HMR
```

**Don't edit these files.** If you have issues, delete the folder and restart.

## Configuration Files

### `warlock.config.ts`

Controls Warlock-specific settings:

```typescript title="warlock.config.ts"
import { defineConfig } from "@warlock.js/core";
import { authMigrations } from "@warlock.js/auth";

export default defineConfig({
  cli: {
    commands: [
      // Register custom CLI commands
    ],
  },
  database: {
    migrations: authMigrations,
  },
});
```

### `.env`

Environment variables. Keep secrets here:

```env title=".env"
# App
APP_NAME=myapp
LOCALE_CODE=en
TIMEZONE=UTC

# HTTP
HTTP_PORT=2030
HTTP_HOST=127.0.0.1
BASE_URL=http://${HTTP_HOST}:${HTTP_PORT}

# Database
DB_HOST=127.0.0.1
DB_PORT=27017
DB_NAME=myapp
DB_USERNAME=root
DB_PASSWORD=root

# Cache
CACHE_DRIVER=memory
REDIS_HOST=127.0.0.1
REDIS_PORT=6379

# Mail
MAIL_HOST=
MAIL_PORT=465
MAIL_USERNAME=
MAIL_PASSWORD=
```

See [Environment & Config](./env-config) for details.

## File Naming Conventions

| Type       | Pattern           | Example                     |
| ---------- | ----------------- | --------------------------- |
| Controller | `*.controller.ts` | `create-post.controller.ts` |
| Model      | `*.model.ts`      | `post.model.ts`             |
| Repository | `*.repository.ts` | `posts.repository.ts`       |
| Service    | `*.service.ts`    | `post.service.ts`           |
| Event      | `*.event.ts`      | `post-created.event.ts`     |
| Resource   | `*.resource.ts`   | `post.resource.ts`          |
| Command    | `*.command.ts`    | `create-admin.command.ts`   |
| Seeder     | `*.seeder.ts`     | `users.seeder.ts`           |
| Routes     | `routes.ts`       | `routes.ts`                 |
