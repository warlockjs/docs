# 🌐 Introduction

HTTP requests are the backbone of any web application. Warlock.js provides a simple and intuitive API to handle HTTP requests efficiently.

## 🚀 Basic Usage

Following the principles of [Express](https://expressjs.com/) and [Fastify](https://www.fastify.io/), Warlock.js employs the concept of `request` and `response` objects.

When a request is made, Warlock.js creates a `request` object containing all the information about the request, and a `response` object containing all the information about the response.

Let's see a simple example:

```ts title="src/app/shared/routes.ts"
import { router, type Request, type Response } from "@warlock.js/core";

router.get("/", (request: Request, response: Response) => {
  return response.success({
    message: "Hello World",
  });
});
```

In this example, we created a simple route that returns a success response with the message `Hello World`.

## 🔄 How it Works

We define our [app routes](./routes) to specify which controllers will be executed when a request is made to a specific route.

When a request is made, Warlock.js checks if there is a route that matches the request URL. If a match is found, it executes the controller and returns the response.

## 🛡️ Middleware

Middleware is a function executed before the controller. It's used to modify the request or response, or to execute some code before the controller.

The HTTP cycle can be defined as follows:

1. Define the routes
2. Execute the middleware (if it exists)
3. Execute the controller that receives the request and returns the response
4. Send the response

## 🤔 Handler or Controller?

Both terms are interchangeable in Warlock.js. However, we primarily use the term **Controller** to align with the directory structure (`src/app/.../controllers`) and common industry conventions.

## 🆚 Function Controllers vs. Class Controllers

In TypeScript, you can use both paradigms. Let's explore the differences.

### Using Function Controllers

Function controllers are ideal for simple routes that perform straightforward tasks. They are perfect for decoupling code, such as a `login` route where you just need to validate the request and return the response.

## Using Class Controllers

In Warlock.js, we prefer using `classes` for `RESTful` APIs, as it makes the code easier to maintain and extend with additional features.

That's why we [introduced Restful Classes](./restful) to simplify the creation of a `RESTful` API with around six routes in a single class.

## 🔄 HTTP Request Life Cycle

The HTTP request life cycle in Warlock.js can be outlined as follows:

1.  **Request Initiation**: The router matches the incoming request URL to a registered route.
2.  **Context Initialization**: The framework creates a request store context, initializing request and response objects.
3.  **Middleware Execution**: Global and route-specific middleware are executed in order.
4.  **Validation Cycle**: If a `validation` property exists on the controller, the request data is validated. Failure here returns a 422 response.
5.  **Pre-Action Event**: The `executingAction` event is triggered.
6.  **Controller Execution**: The controller function is executed to process the request.
7.  **Post-Action Event**: The `executedAction` event is triggered after successful execution.
8.  **Response Finalization**: The controller's return value is sent back to the client.

By understanding these steps, developers can effectively manage the flow of data and control the behavior of their web applications using Warlock.js.
