## Connection options reference

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `driver` | `"rabbitmq"` | — | **Required.** Selects the RabbitMQ driver. |
| `host` | `string` | `"localhost"` | Broker hostname or IP address. |
| `port` | `number` | `5672` | AMQP protocol port (`5671` for TLS). |
| `username` | `string` | `"guest"` | Authentication username. |
| `password` | `string` | `"guest"` | Authentication password. |
| `vhost` | `string` | `"/"` | Virtual host path (namespace isolation). |
| `uri` | `string` | — | Full AMQP URI. Overrides host/port/credentials if provided. |
| `heartbeat` | `number` | `60` | Heartbeat interval in seconds. Detects stale connections. |
| `connectionTimeout` | `number` | `30000` | Connection timeout in milliseconds. |
| `reconnect` | `boolean` | `true` | Automatically reconnect on disconnect. |
| `reconnectDelay` | `number` | `5000` | Delay between reconnect attempts in milliseconds. |
| `prefetch` | `number` | — | Global prefetch count (max unacked messages per consumer). |
| `name` | `string` | `"default"` | Broker name in the registry (for multi-broker setups). |
| `isDefault` | `boolean` | `true` | Whether this is the default broker when calling `herald()`. |
| `clientOptions` | `RabbitMQClientOptions` | — | Native amqplib options (frameMax, channelMax, socket). |

## Basic connection

```typescript title="src/main.ts"
import { connectToBroker } from '@warlock.js/herald';

await connectToBroker({
  driver: 'rabbitmq',
  host: 'rabbitmq.local',
  port: 5672,
  username: 'app',
  password: 'secret123',
  vhost: '/orders',
});
```

## URI-based connection (recommended for production)

Use a full AMQP URI to avoid exposing credentials in source code:

```typescript title="src/main.ts"
import { connectToBroker } from '@warlock.js/herald';

await connectToBroker({
  driver: 'rabbitmq',
  uri: process.env.RABBITMQ_URI,
  // e.g., amqp://app:secret123@rabbitmq.internal:5672/orders
});
```

:::tip
Store the full URI in an environment variable. The URI format is:
`amqp://[username[:password]@]host[:port][/vhost]`
:::

## TLS/SSL connection

For encrypted connections (required in production), use TLS with client certificates:

```typescript title="src/main.ts"
import { connectToBroker } from '@warlock.js/herald';
import fs from 'fs';

await connectToBroker({
  driver: 'rabbitmq',
  host: 'rabbitmq.internal',
  port: 5671, // TLS port
  username: process.env.RABBITMQ_USER,
  password: process.env.RABBITMQ_PASS,
  clientOptions: {
    socket: {
      ca: [fs.readFileSync('/certs/ca.crt', 'utf-8')],
      cert: fs.readFileSync('/certs/client.crt', 'utf-8'),
      key: fs.readFileSync('/certs/client.key', 'utf-8'),
      passphrase: process.env.RABBITMQ_KEY_PASSPHRASE,
      rejectUnauthorized: true,
    },
  },
});
```

Or load certificates from environment variables (common in containerized deployments):

```typescript title="src/main.ts"
await connectToBroker({
  driver: 'rabbitmq',
  uri: process.env.RABBITMQ_URI, // amqps:// for secure URI
  clientOptions: {
    socket: {
      ca: process.env.RABBITMQ_CA_CERT ? [process.env.RABBITMQ_CA_CERT] : undefined,
      cert: process.env.RABBITMQ_CLIENT_CERT,
      key: process.env.RABBITMQ_CLIENT_KEY,
      rejectUnauthorized: !process.env.ALLOW_SELF_SIGNED,
    },
  },
});
```

:::danger
Always use TLS in production. Plain AMQP sends credentials in cleartext, making your system vulnerable to credential theft and man-in-the-middle attacks.
:::

## Heartbeat and connection health

The heartbeat mechanism detects stale connections. Both client and server send heartbeat frames at regular intervals:

```typescript title="src/main.ts"
await connectToBroker({
  driver: 'rabbitmq',
  host: 'localhost',
  heartbeat: 60, // Check every 60 seconds (default)
});
```

If no traffic occurs for `heartbeat` seconds, the client sends a heartbeat frame. If the server doesn't respond within twice the heartbeat interval, Herald closes the connection and triggers reconnection.

| Value | Use case |
|-------|----------|
| `60` (default) | Good balance for most applications |
| `30` | Latency-sensitive applications that need faster detection |
| `0` | Disable — only in tests, never in production |

## Auto-reconnect

When the connection drops, Herald automatically reconnects if `reconnect: true`:

```typescript title="src/main.ts"
import { connectToBroker, brokerRegistry, herald } from '@warlock.js/herald';

await connectToBroker({
  driver: 'rabbitmq',
  host: 'localhost',
  reconnect: true,
  reconnectDelay: 5000,
});

brokerRegistry.on('connected', (broker) => {
  console.log(`${broker.name} connected`);
});

brokerRegistry.on('disconnected', (broker) => {
  console.warn(`${broker.name} disconnected`);
});

herald().driver.on('reconnecting', (attempt) => {
  console.log(`Reconnection attempt #${attempt}`);
});
```

:::tip
Disable auto-reconnect in tests with `reconnect: false` to avoid hanging connections after the test suite finishes.
:::

## Prefetch and QoS

The prefetch count controls how many unacknowledged messages a consumer holds at once.

### Global prefetch

```typescript title="src/main.ts"
await connectToBroker({
  driver: 'rabbitmq',
  host: 'localhost',
  prefetch: 10,
});
```

### Per-subscriber prefetch

```typescript title="src/consumers/setup.ts"
import { herald } from '@warlock.js/herald';

// High throughput for lightweight analytics events
herald().channel('order.placed').subscribe(
  async (message, ctx) => {
    await trackOrderEvent(message.payload);
    await ctx.ack();
  },
  { prefetch: 50 },
);

// Process one payment at a time — no margin for errors
herald().channel('payment.processed').subscribe(
  async (message, ctx) => {
    await processPayment(message.payload);
    await ctx.ack();
  },
  { prefetch: 1 },
);
```

| Scenario | Recommended prefetch |
|----------|---------------------|
| Fast, lightweight processing | 10–50 |
| Slow or heavy processing | 1 |
| Mixed workload | Start at 5–10, tune from monitoring |

## Virtual hosts (vhost)

Virtual hosts provide namespace isolation within a single RabbitMQ instance — their own queues, exchanges, user permissions, and routing rules:

```typescript title="src/main.ts"
// Local development
await connectToBroker({
  driver: 'rabbitmq',
  host: 'localhost',
  vhost: '/dev',
  username: 'dev_user',
  password: 'dev_pass',
});

// Production (via URI — vhost is the path segment)
await connectToBroker({
  driver: 'rabbitmq',
  uri: process.env.RABBITMQ_URI, // amqp://...@host/production
});
```

Use virtual hosts to safely share one RabbitMQ cluster across dev, staging, and production without data leakage or queue name collisions.

## `RabbitMQClientOptions`

Protocol-level options passed to amqplib as the `clientOptions` field on `RabbitMQConnectionOptions`.

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `frameMax` | `number` | `0` | Maximum AMQP frame size in bytes. `0` = unlimited. Increase for large messages. |
| `channelMax` | `number` | `0` | Maximum number of channels per connection. `0` = unlimited. |
| `locale` | `string` | `"en_US"` | Client locale preference for broker error messages. |
| `socket` | `RabbitMQSocketOptions` | — | Socket-level TLS and connection configuration. |

```typescript title="src/main.ts"
await connectToBroker({
  driver: 'rabbitmq',
  host: 'localhost',
  clientOptions: {
    frameMax: 131072,
    channelMax: 2048,
    locale: 'en_US',
  },
});
```

## `RabbitMQSocketOptions`

Socket-level options passed to `clientOptions.socket` on `RabbitMQConnectionOptions`. Controls TLS certificates, TCP behaviour, and connection timeouts.

### TLS options

| Option | Type | Description |
|--------|------|-------------|
| `ca` | `string \| Buffer \| Array<string \| Buffer>` | Certificate authority for verifying the server. |
| `cert` | `string \| Buffer` | Client certificate for mutual TLS (mTLS). |
| `key` | `string \| Buffer` | Client private key. |
| `passphrase` | `string` | Passphrase to decrypt an encrypted private key. |
| `servername` | `string` | SNI hostname for TLS virtual hosting. |
| `rejectUnauthorized` | `boolean` | Validate the server certificate chain. Set `false` only in dev/test. |

### TCP options

| Option | Type | Description |
|--------|------|-------------|
| `keepAlive` | `boolean \| number` | Enable TCP keep-alive probes to detect dead connections. Pass a number to set the initial delay in milliseconds. |
| `noDelay` | `boolean` | Disable Nagle's algorithm — send data immediately (reduces latency). |
| `timeout` | `number` | Socket timeout in milliseconds. |

### Minimal mTLS example

```typescript title="src/main.ts"
import { connectToBroker } from '@warlock.js/herald';
import fs from 'fs';

await connectToBroker({
  driver: "rabbitmq",
  host: "rabbitmq.example.com",
  port: 5671,
  clientOptions: {
    socket: {
      ca: fs.readFileSync("ca.pem"),
      cert: fs.readFileSync("client.pem"),
      key: fs.readFileSync("client-key.pem"),
      rejectUnauthorized: true,
    },
  },
});
```

## Delayed messages

Delivering a message after a delay requires the [rabbitmq-delayed-message-exchange](https://github.com/rabbitmq/rabbitmq-delayed-message-exchange) plugin:

```typescript title="src/services/order.service.ts"
import { heraldChannel } from '@warlock.js/herald';

// Send a follow-up reminder 24 hours after order placement
await heraldChannel('order.reminder').publish(
  { orderId: 'ord-123', userId: 'usr-456' },
  { delay: 24 * 60 * 60 * 1000 }, // 24 hours in ms
);
```

:::info
Without the plugin enabled on your broker, the `delay` option is silently ignored and the message is delivered immediately.
:::

## Priority queues

Messages support a priority from `0` (lowest) to `255` (highest). Higher-priority messages are processed first:

```typescript title="src/services/notification.service.ts"
import { heraldChannel } from '@warlock.js/herald';

// Urgent: account security alert
await heraldChannel('notification.send').publish(
  { userId: 'usr-456', type: 'security-alert', message: 'New login detected' },
  { priority: 10 },
);

// Low priority: weekly digest
await heraldChannel('notification.send').publish(
  { userId: 'usr-456', type: 'digest', message: 'Your weekly summary' },
  { priority: 1 },
);
```

## Docker Compose for local development

```yaml title="docker-compose.yml"
services:
  rabbitmq:
    image: rabbitmq:3.13-management
    container_name: rabbitmq
    ports:
      - "5672:5672"
      - "15672:15672"    # Management UI at http://localhost:15672
    environment:
      RABBITMQ_DEFAULT_USER: admin
      RABBITMQ_DEFAULT_PASS: admin123
      RABBITMQ_DEFAULT_VHOST: /
    volumes:
      - rabbitmq_data:/var/lib/rabbitmq
    healthcheck:
      test: ["CMD", "rabbitmq-diagnostics", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  rabbitmq_data:
```

Connect using the service name as the host:

```typescript title="src/main.ts"
await connectToBroker({
  driver: 'rabbitmq',
  host: process.env.RABBITMQ_HOST ?? 'localhost',
  port: Number(process.env.RABBITMQ_PORT ?? 5672),
  username: process.env.RABBITMQ_USER ?? 'admin',
  password: process.env.RABBITMQ_PASS ?? 'admin123',
});
```

## Health checking

```typescript title="src/health/broker.health.ts"
import { herald } from '@warlock.js/herald';

export async function getBrokerHealth() {
  try {
    const result = await herald().driver.healthCheck?.();
    return {
      status: result?.healthy ? 'healthy' : 'unhealthy',
      latency: result?.latency,
      broker: herald().name,
    };
  } catch (error) {
    return { status: 'unhealthy', error: error.message };
  }
}
```
