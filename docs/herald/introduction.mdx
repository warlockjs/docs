Herald is a typed, event-driven messaging library for Node.js that lets you decouple services in your application by routing events and messages through a broker. It handles the complexity of connecting to message brokers (like RabbitMQ), defining typed events, publishing them, and subscribing to them with type safety.

## What Herald does

Herald wraps a message broker in a clean, JavaScript-friendly API. You define typed events, publish them from one service, and handle them in another—all while keeping your code loosely coupled and testable.

```ts title="src/main.ts"
import { connectToBroker, publishEvent } from '@warlock.js/herald';

// Connect at startup
await connectToBroker({ driver: 'rabbitmq', host: 'localhost' });

// Publish an event
await publishEvent(new OrderPlaced({ orderId: '123', total: 99.99 }));

// In another service, handle it
import { subscribeConsumer } from '@warlock.js/herald';

subscribeConsumer(SendOrderNotificationConsumer);
```

## Core concepts at a glance

| Concept | What it is |
|---------|-----------|
| **Broker** | The central connection to your message broker (RabbitMQ, Kafka, etc.). Each named broker is a separate connection. |
| **Driver** | The adapter that speaks to a specific broker technology. Same code, different drivers = different brokers. |
| **Channel** | A named topic or queue where events flow. Automatically created and cached. Think of it as a pub/sub topic. |
| **EventMessage** | A typed event class that producers send. Carries a payload and metadata (ID, timestamp, etc.). |
| **EventConsumer** | A typed handler class that processes events from a channel. Knows which event it handles and how. |
| **@Consumable** | A decorator that auto-registers a consumer when the broker connects, so you don't manually wire things up. |

## Supported drivers

| Driver | Status | Notes |
|--------|--------|-------|
| **RabbitMQ** | Stable | Production-ready. Requires `amqplib` peer dependency. |
| **Kafka** | Coming soon | High-throughput distributed events. |
| **Redis Streams** | Planned | Lightweight, in-memory broker. |
| **SQS** | Planned | AWS-native queue service. |

## When to use Herald

- **Microservices**: Decouple OrderService from PaymentService from NotificationService.
- **Async workflows**: Offload heavy work (emails, reports) to background consumers.
- **Event sourcing**: Build an audit trail of what happened in your system.
- **Cross-service communication**: Avoid direct HTTP calls; let events carry the data.
- **Scalability**: Add consumer instances without changing producer code.

    ```bash
    pnpm add @warlock.js/herald amqplib
    ```
  </TabItem>

The `amqplib` package is required if you're using RabbitMQ as your broker driver. If you plan to use a different driver later, you'll install its peer dependency instead.
