# Getting Started

## Installation

```bash
npm install @warlock.js/herald amqplib
```

## Connect to a broker

Call `connectToCommunicator()` once during your application bootstrap. It creates a driver, establishes the connection, and registers the communicator under a name so the rest of your code can retrieve it anywhere.

```typescript title="src/bootstrap.ts"
import { connectToCommunicator } from "@warlock.js/herald";

await connectToCommunicator({
  driver: "rabbitmq",
  host: process.env.RABBITMQ_HOST ?? "localhost",
  port: 5432,
  username: process.env.RABBITMQ_USER ?? "guest",
  password: process.env.RABBITMQ_PASS ?? "guest",
  vhost: "/",
});
```

`connectToCommunicator` returns the `Communicator` instance, which you can use immediately, but you normally won't need to hold onto the return value — just call `communicators()` wherever you need it.

## Publish your first message

```typescript
import { communicators } from "@warlock.js/herald";

await communicators().channel("user.created").publish({ userId: 1, email: "alice@example.com" });
```

## Subscribe to messages

```typescript
import { communicators } from "@warlock.js/herald";

communicators()
  .channel<{ userId: number; email: string }>("user.created")
  .subscribe(async (message, ctx) => {
    console.log("New user:", message.payload);
    await ctx.ack();
  });
```

The `ctx` object gives you full control over acknowledgement:

| Method | Effect |
|---|---|
| `ctx.ack()` | Mark the message as successfully processed |
| `ctx.nack(requeue?)` | Negative-acknowledge; requeue by default |
| `ctx.reject()` | Reject without requeue |
| `ctx.retry(delay?)` | Re-publish with an incremented retry counter |
| `ctx.reply(payload)` | Reply to an RPC request |

:::tip Smart auto-ack
If your handler returns normally without calling any ack/nack method, Herald auto-acks the message. If your handler throws, Herald auto-nacks it. You only need explicit calls when you want non-default behaviour.
:::

## Multiple communicators

Register each communicator with a unique name, and mark exactly one as `isDefault`.

```typescript
import { connectToCommunicator } from "@warlock.js/herald";

// Primary broker — becomes default automatically
await connectToCommunicator({
  driver: "rabbitmq",
  name: "primary",
  isDefault: true,
  host: process.env.RABBITMQ_HOST,
});

// Analytics broker
await connectToCommunicator({
  driver: "rabbitmq",
  name: "analytics",
  host: process.env.ANALYTICS_RABBITMQ_HOST,
});
```

Then address them by name:

```typescript
import { communicators } from "@warlock.js/herald";

// Uses the default communicator
communicators().channel("notifications").publish({ ... });

// Uses the "analytics" communicator
communicators("analytics").channel("page.viewed").publish({ page: "/home" });
```

## Using the communicatorChannel shortcut

If you always use the default communicator you can skip the intermediate `communicators()` call:

```typescript
import { communicatorChannel } from "@warlock.js/herald";

const channel = communicatorChannel<{ userId: number }>("user.created");
await channel.publish({ userId: 42 });
```

## Health check

```typescript
import { communicators } from "@warlock.js/herald";

const result = await communicators().healthCheck();

if (result.healthy) {
  console.log(`Healthy — latency ${result.latency}ms`);
} else {
  console.error(`Unhealthy: ${result.error}`);
}
```
