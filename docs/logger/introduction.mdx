# Logger

> Structured, multi-channel logging for Node.js — with five severity levels, a clean three-argument API, and full TypeScript support.

## Installation

```bash
npm install @warlock.js/logger
```

## Zero to first log

:::info No channels = no output
The logger starts with **no channels** — nothing is printed or written until you add at least one.
:::

```ts title="src/index.ts"
import { logger, log } from "@warlock.js/logger";
import { ConsoleLog } from "@warlock.js/logger";

logger.addChannel(new ConsoleLog());

await log.info("users", "register", "New user created successfully");
```

That's it. Three lines of setup, then you're logging.

## The module · action · message pattern

Every log call carries three pieces of context: **module**, **action**, and **message**.

```ts
await log.info("users", "register", "New user created");
//            ^^^^^^^ ^^^^^^^^^^ ^^^^^^^^^^^^^^^^^^^^^^
//            module   action      message
```

This trio lets you filter and search logs by feature area — no more wading through flat text. Here are real-world examples across domains:

```ts
// Auth
await log.info("auth", "login", "User signed in");
await log.warn("auth", "tokenExpired", "JWT expired for session");

// Payments
await log.info("payments", "chargeInitiated", { amount: 4999, currency: "USD" });
await log.error("payments", "chargeFailed", new Error("Stripe declined card"));

// Email
await log.success("email", "welcomeSent", "Welcome email delivered");
await log.warn("email", "bounced", "Delivery failed — invalid address");

// Orders
await log.debug("orders", "priceCalculation", { subtotal: 120, discount: 10 });
await log.success("orders", "fulfillment", "Order shipped via FedEx");
```

:::tip
Use lowercase `camelCase` for both `module` and `action` — they become searchable keys in file and JSON channels.
:::

## The five log levels

| Level | Method | When to use | Console style |
|---|---|---|---|
| `debug` | `log.debug()` | Dev diagnostics, variable dumps, flow tracing | dim grey |
| `info` | `log.info()` | Normal operational events (user registered, job queued) | blue |
| `warn` | `log.warn()` | Unexpected but recoverable conditions (retry, fallback) | yellow |
| `error` | `log.error()` | Failures that need attention; accepts `Error` objects | red |
| `success` | `log.success()` | Explicit happy-path confirmation (payment completed) | green |

```ts
await log.debug("cache", "miss", "Key not found, fetching from DB");
await log.info("jobs", "queued", "SendInvoice job added to queue");
await log.warn("api", "rateLimitApproaching", "80% of quota used");
await log.error("db", "connectionFailed", new Error("ECONNREFUSED 5432"));
await log.success("payments", "captured", "Payment of $49.99 captured");
```

## Structured logging

When you need to attach arbitrary metadata, pass a `LoggingData` object instead of the three-argument shorthand.

```ts
import { log, type LoggingData } from "@warlock.js/logger";

const entry: LoggingData = {
  type: "error",
  module: "orders",
  action: "checkout",
  message: "Card declined",
  context: {
    orderId: "ord_9f2a",
    userId: 101,
    amount: 4999,
    retryCount: 2,
  },
};

await log(entry);
```

The `LoggingData` type in full:

```ts
type LoggingData = {
  type: "info" | "debug" | "warn" | "error" | "success";
  module: string;
  action: string;
  message: any;
  context?: Record<string, any>;
};
```

**Use the shorthand** for simple messages. **Use `LoggingData`** when you need the `context` field — file and JSON channels persist it alongside the entry; the console channel displays it inline.

## Logging errors

Pass an `Error` instance directly as the `message` argument. File channels capture the full stack trace automatically.

```ts
try {
  await processPayment(orderId);
} catch (err) {
  await log.error("payments", "processPayment", err);
}
```

:::warning
Always pass the raw `Error` object — do **not** stringify it first (`err.message`). Passing the object lets file channels record the full stack trace.
:::

## `log` vs `logger`

| Export | Type | Typical use |
|---|---|---|
| `log` | Bound shorthand object | Everyday logging — `log.info(...)`, `log.error(...)` |
| `logger` | `Logger` class instance | One-time setup — `logger.addChannel(...)`, `logger.configure(...)` |

Both target the same underlying instance. Calling `log.info(...)` is identical to calling `logger.info(...)`.

```ts
import { log, logger } from "@warlock.js/logger";

// These are equivalent:
await log.info("users", "login", "User signed in");
await logger.info("users", "login", "User signed in");

// Only logger exposes the setup API:
logger.addChannel(new ConsoleLog());
logger.configure({ levels: ["warn", "error"] });
logger.setChannels([new ConsoleLog(), new FileLog()]);
```

The `log` object binds exactly: `info`, `debug`, `warn`, `error`, `success`, and `channel`.

## Built-in channels

| Channel | Name | What it does |
|---|---|---|
| `ConsoleLog` | `"console"` | Colorized terminal output, one line per entry |
| `FileLog` | `"file"` | Plain-text `.log` files with `"single"`, `"daily"`, or `"hourly"` rotation |
| `JSONFileLog` | `"fileJson"` | Structured `.json` files — ideal for log aggregators |

See the [Channels](./channels) page for full configuration options and examples.

## Utilities

### `captureAnyUnhandledRejection()`

Registers process-level handlers for `unhandledRejection` and `uncaughtException`. Both are forwarded to `log.error()` with `module: "app"`. Call it once at startup.

```ts title="src/index.ts"
import { logger, log, captureAnyUnhandledRejection } from "@warlock.js/logger";
import { ConsoleLog } from "@warlock.js/logger";

logger.addChannel(new ConsoleLog());
captureAnyUnhandledRejection();

// From here, unhandled promise rejections and uncaught exceptions
// are automatically logged before the process exits.
```

### `clearMessage()`

Strips ANSI escape codes from a string. Useful when you need clean output for non-terminal destinations.

```ts
import { clearMessage } from "@warlock.js/logger";

clearMessage("\u001b[32mHello\u001b[0m"); // → "Hello"
```
