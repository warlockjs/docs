# ūüĒĘ ID Generation Internals

Cascade's auto-increment ID system is built on a clean contract-and-implementation layered architecture. Instead of a single monolithic class, ID generation is handled by the `IdGeneratorContract` interface and its concrete implementations (such as `MongoIdGenerator`). Models expose a high-level `generateNextId()` instance method that delegates all the heavy lifting to the configured generator. ūü™Ą

## ūüö¶ How It Works

1. Each model has a configured **ID generator** (set at the driver level).
2. When a new document is saved, the model calls `generateNextId()` internally.
3. The generator atomically increments a counter stored in a dedicated **counter collection** (e.g. `counters`) and returns the next integer.
4. The returned integer becomes the document's `id`.

:::tip
For SQL people, think of the counter collection as the equivalent of an auto-increment sequence or `INFORMATION_SCHEMA` sequence tracker ‚ÄĒ it stores the last-used ID per collection.
:::

---

## ūüďĄ `GenerateIdOptions` Type

Options passed to the generator whenever a new ID is needed.

```ts
type GenerateIdOptions = {
  /** Target collection / table name. */
  table: string;
  /** Starting value for the very first record. Defaults to 1. */
  initialId?: number;
  /** Step between consecutive IDs. Defaults to 1. */
  incrementIdBy?: number;
};
```

## ūü§Ě `IdGeneratorContract`

The interface every ID generator must implement. Import it from `@warlock.js/cascade` when writing a custom generator.

```ts
import type { IdGeneratorContract, GenerateIdOptions } from "@warlock.js/cascade";
```

### Methods

| Method | Signature | Description |
|---|---|---|
| `generateNextId` | `(options: GenerateIdOptions) => Promise<number>` | Atomically increments the counter and returns the next ID. Throws on concurrent write failure. |
| `getLastId` | `(table: string) => Promise<number>` | Returns the last generated ID for the given collection. Returns `0` if no IDs have been generated yet. |
| `setLastId` | `(table: string, id: number) => Promise<void>` | Overwrites the stored counter for a collection. Useful for migrations or data repairs. |

### Writing a Custom Generator

```ts title="src/my-id-generator.ts"
import type { IdGeneratorContract, GenerateIdOptions } from "@warlock.js/cascade";

export class MyIdGenerator implements IdGeneratorContract {
  async generateNextId(options: GenerateIdOptions): Promise<number> {
    const { table, initialId = 1, incrementIdBy = 1 } = options;
    // your atomic increment logic here ‚Ä¶
    return nextId;
  }

  async getLastId(table: string): Promise<number> {
    // read from your counter store ‚Ä¶
    return lastId;
  }

  async setLastId(table: string, id: number): Promise<void> {
    // write to your counter store ‚Ä¶
  }
}
```

## ūüćÉ `MongoIdGenerator`

The built-in MongoDB implementation of `IdGeneratorContract`. It stores one counter document per collection inside a dedicated **counter collection** (default name: `counters`).

```ts
import { MongoIdGenerator } from "@warlock.js/cascade";
```

### Constructor

```ts
new MongoIdGenerator(driver: MongoDbDriver, counterCollection?: string)
```

| Parameter | Type | Default | Description |
|---|---|---|---|
| `driver` | `MongoDbDriver` | ‚ÄĒ | The active MongoDB driver instance. |
| `counterCollection` | `string` | `"counters"` | Name of the collection used to store ID counters. |

### How It Stores Counters

Each document in the counter collection looks like:

```json
{ "_id": "categories", "seq": 42 }
```

`generateNextId` uses MongoDB's `findOneAndUpdate` with `$inc` and `upsert: true` to guarantee atomic, collision-free increments even under high concurrency.

### Reading and Writing Counters Directly

:::info
You should rarely need to call these directly ‚ÄĒ the model handles ID generation automatically. Use them only for migrations, data repair, or admin tooling.
:::

```ts title="src/scripts/repair-counters.ts"
import { MongoIdGenerator } from "@warlock.js/cascade";
import { connection } from "./db";

const generator = new MongoIdGenerator(connection.driver);

// Read the current counter for a collection
const last = await generator.getLastId("categories");
console.log(last); // e.g. 42

// Manually set the counter (e.g. after a bulk import)
await generator.setLastId("categories", 1000);

// Generate the next ID (increments the counter atomically)
const next = await generator.generateNextId({
  table: "categories",
  initialId: 1,
  incrementIdBy: 1,
});
console.log(next); // 1001
```

## ūüŹ∑ÔłŹ `model.generateNextId()`

The model-level method that orchestrates ID generation. It reads the model's static configuration (`initialId`, `incrementIdBy`) and delegates to the driver's configured `IdGeneratorContract` implementation.

```ts title="src/models/category.ts"
import { Model } from "@warlock.js/cascade";

export class Category extends Model {
  public static table = "categories";
}

// Inside a custom save flow:
const category = new Category();
await category.generateNextId();
console.log(category.id); // e.g. 43
```

:::tip
You almost never call `generateNextId()` manually. Cascade's `save()` method calls it automatically whenever `autoGenerateId` is `true` (the default) and the document has no existing `id`.
:::

### Relevant Static Model Properties

These statics influence what `generateNextId()` passes to the underlying generator:

| Property | Type | Default | Description |
|---|---|---|---|
| `autoGenerateId` | `boolean` | `true` | Whether to auto-generate an ID on save. |
| `initialId` | `number` | `1` | The very first ID value for a new collection. |
| `incrementIdBy` | `number` | `1` | Step between consecutive IDs. |
| `randomInitialId` | `boolean` | `false` | Start from a random initial ID. |
| `randomIncrement` | `boolean` | `false` | Use a random increment step. |

---

## ‚ö†ÔłŹ Best Practices

- **Do not** call `setLastId` while active traffic is writing to the collection ‚ÄĒ always drain writes first.
- **Do** use `setLastId` after bulk imports to advance the counter past the highest imported ID, preventing future collisions.
- **Do not** share a counter collection across databases that may have conflicting collection names ‚ÄĒ use a unique `counterCollection` name per database context.

:::info
All ID generation is atomic at the database level. Even under high concurrency, no two documents will receive the same auto-increment ID for the same collection.
:::
