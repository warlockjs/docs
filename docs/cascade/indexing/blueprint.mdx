# Blueprint

Cascade exposes **blueprint** classes for inspecting the live schema of a connected database — listing tables, columns, and indexes. Blueprint is a **read-only introspection tool**; if you need to create or drop indexes, do that inside a [migration](../migrations/overview).

:::info Blueprint is not a factory
`new Blueprint("users")` and the `blueprint()` factory function are **no longer part of the API**. Use the driver-specific blueprint classes described below instead.
:::

## `DriverBlueprintContract`

Both the MongoDB and PostgreSQL blueprint classes implement the same interface so you can write driver-agnostic introspection code.

```ts
import { DriverBlueprintContract } from "@warlock.js/cascade";
```

### Interface methods

| Method | Returns | Description |
|---|---|---|
| `listTables()` | `Promise<string[]>` | All table / collection names in the database |
| `listIndexes(table)` | `Promise<TableIndexInformation[]>` | All indexes on the given table / collection |
| `listColumns(table)` | `Promise<string[]>` | Column names for the given table |
| `tableExists(table)` | `Promise<boolean>` | Whether the table / collection exists |

### `TableIndexInformation` type

Each entry returned by `listIndexes` conforms to this shape:

```ts
type TableIndexInformation = {
  name: string;
  columns?: string[];
  type?: string;
  unique: boolean;
  partial: boolean;
  options: Record<string, any>;
};
```

- **`name`** — the index name as stored in the database
- **`columns`** — column(s) covered by the index (may be empty for MongoDB schemaless collections)
- **`type`** — index type string (e.g. `"btree"`, `"text"`, `"2dsphere"`)
- **`unique`** — whether the index enforces uniqueness
- **`partial`** — whether the index is a partial / sparse index
- **`options`** — raw driver-specific options object

## `MongoDBBlueprint`

`MongoDBBlueprint` provides collection and index introspection for a MongoDB connection. It implements `DriverBlueprintContract`.

### Getting a `MongoDBBlueprint` via the driver

```ts
import { connection } from "@warlock.js/cascade";

// Obtain the underlying Db instance from the active connection
const db = connection().database; // Db (from the `mongodb` package)

import { MongoDBBlueprint } from "@warlock.js/cascade";

const blueprint = new MongoDBBlueprint(db);
```

### Available methods

#### `listTables()`

Returns all collection names in the connected database.

```ts
const collections = await blueprint.listTables();
// ["users", "orders", "products"]
```

#### `listIndexes(table)`

Returns index metadata for a collection.

```ts
const indexes = await blueprint.listIndexes("users");
/*
[
  { name: "_id_", columns: ["_id"], unique: false, partial: false, options: {} },
  { name: "email_1", columns: ["email"], unique: true, partial: false, options: {} },
]
*/
```

#### `tableExists(table)`

Checks whether a collection exists.

```ts
const exists = await blueprint.tableExists("users");
// true
```

#### `listColumns(table)`

:::info MongoDB is schemaless
MongoDB has no fixed schema, so `listColumns()` always returns an empty array and logs a warning. Use your model definitions as the source of truth for field names.
:::

```ts
const columns = await blueprint.listColumns("users");
// [] — always empty for MongoDB
```

## `PostgresBlueprint`

`PostgresBlueprint` provides table and index introspection for a PostgreSQL connection using `information_schema` and `pg_catalog`. It implements `DriverBlueprintContract`.

### Getting a `PostgresBlueprint` via the driver

```ts
import { PostgresDriver, PostgresBlueprint } from "@warlock.js/cascade";

// driver is an already-configured PostgresDriver instance
const blueprint = new PostgresBlueprint(driver);
```

### Available methods

#### `listTables()`

Lists all public base table names in the connected database.

```ts
const tables = await blueprint.listTables();
// ["users", "orders", "products"]
```

#### `listIndexes(table)`

Lists and parses all indexes on the given table.

```ts
const indexes = await blueprint.listIndexes("users");
/*
[
  {
    name: "users_pkey",
    columns: ["id"],
    type: "btree",
    unique: true,
    partial: false,
    options: {},
  },
  {
    name: "users_email_unique",
    columns: ["email"],
    type: "btree",
    unique: true,
    partial: false,
    options: {},
  },
]
*/
```

#### `listColumns(table)`

Lists column names ordered by their ordinal position.

```ts
const columns = await blueprint.listColumns("users");
// ["id", "name", "email", "created_at"]
```

#### `tableExists(table)`

Checks whether the table exists in the public schema.

```ts
const exists = await blueprint.tableExists("users");
// true
```

## Writing driver-agnostic introspection code

Because both blueprints implement `DriverBlueprintContract`, you can pass either one to shared utilities:

```ts
import { DriverBlueprintContract } from "@warlock.js/cascade";

async function printIndexes(
  blueprint: DriverBlueprintContract,
  table: string,
) {
  const indexes = await blueprint.listIndexes(table);

  for (const index of indexes) {
    const kind = index.unique ? "UNIQUE" : "INDEX";
    const cols = index.columns?.join(", ") ?? "—";
    console.log(`[${kind}] ${index.name} (${cols})`);
  }
}

// Works with MongoDB:
await printIndexes(mongoBlueprint, "users");

// Works with PostgreSQL:
await printIndexes(postgresBlueprint, "users");
```

:::tip Create indexes in migrations
Blueprint is intentionally read-only. To create or drop indexes, use the migration system — this keeps your schema changes versioned, repeatable, and safe to run in CI/CD pipelines.

See [Migrations Overview](../migrations/overview) for details.
:::
