# 🔄 Syncing Models

Syncing models is a major powerful feature in `Cascade`. It lets you embed documents inside other documents and automatically propagate updates whenever the source document changes — including cascading deletes.

## 🧩 The Problem

Say you have a `Category` model and a `Post` model. Each post embeds a category snapshot. The problem: when the category name changes, every embedded copy inside posts stays stale.

## ✅ The Solution

Model sync solves this. When a category is updated, Cascade searches every post that embeds that category and updates the embedded copy automatically.

There are two embedding scenarios:

1. **Single embedded document** — one model embedded as an object
2. **Array of embedded documents** — a list of models embedded as an array

## 🏗️ Setup: `modelSync`

The sync system is powered by the `modelSync` singleton. Import it from `@warlock.js/cascade`:

```ts
import { modelSync } from "@warlock.js/cascade";
```

You register syncs inside a `modelSync.register()` callback. This is the recommended pattern because it is HMR-safe — the callback is re-executed on hot reload and previous subscriptions are automatically cleaned up.

```ts title="src/sync/index.ts"
import { modelSync } from "@warlock.js/cascade";
import { Category } from "../models/category";
import { Post } from "../models/post";

modelSync.register(() => {
  modelSync.sync(Category, Post, "category");
});
```

:::tip
Call `modelSync.register()` once at app startup (e.g. in your bootstrap file). You do not need to put sync logic inside the model class itself.
:::

## 📄 Single Embedded Document

The most common case: a post embeds one category object. When the category is updated, all matching posts are updated.

**Post model** (unchanged — just uses `castModel`):

```ts title="src/models/post.ts"
import { Model, Casts, castModel } from "@warlock.js/cascade";
import { Category } from "./category";

export class Post extends Model {
  public static table = "posts";

  protected casts: Casts = {
    title: "string",
    content: "string",
    category: castModel(Category),
  };
}
```

**Sync registration:**

```ts title="src/sync/index.ts"
import { modelSync } from "@warlock.js/cascade";
import { Category } from "../models/category";
import { Post } from "../models/post";

modelSync.register(() => {
  // When a Category is updated, update the "category" field in Post documents
  modelSync.sync(Category, Post, "category");
});
```

`modelSync.sync(SourceModel, TargetModel, targetField)` creates a single-document sync. Internally it matches by `category.id` inside the `posts` collection.

### Static shortcut

Models expose `Model.sync()` and `Model.syncMany()` as static shortcuts that delegate to `modelSync`:

```ts title="src/sync/index.ts"
import { modelSync } from "@warlock.js/cascade";
import { Category } from "../models/category";
import { Post } from "../models/post";

modelSync.register(() => {
  Category.sync(Post, "category");
});
```

Both `modelSync.sync(Category, Post, "category")` and `Category.sync(Post, "category")` are equivalent.

## 📚 Array of Embedded Documents

When the target field holds an **array** of embedded documents, use `syncMany` instead of `sync`:

```ts title="src/sync/index.ts"
import { modelSync } from "@warlock.js/cascade";
import { Comment } from "../models/comment";
import { Post } from "../models/post";

modelSync.register(() => {
  // When a Comment is updated, update its entry inside the "comments" array in Post
  modelSync.syncMany(Comment, Post, "comments");
  // or equivalently:
  // Comment.syncMany(Post, "comments");
});
```

:::warning
The array-sync example above is illustrative. Storing comments directly inside post documents is generally not recommended — prefer a separate collection.
:::

:::info
All fluent methods shown below work identically for both `sync` and `syncMany` operations.
:::

## 🎨 Embedded Data Shape

By default the sync uses the model's `embedData` property to extract the data to store. You can change this with `embed()`:

```ts title="src/sync/index.ts"
import { modelSync } from "@warlock.js/cascade";
import { Category } from "../models/category";
import { Post } from "../models/post";

modelSync.register(() => {
  modelSync
    .sync(Category, Post, "category")
    .embed("embedMinimal"); // "embedData" | "embedParent" | "embedMinimal"
});
```

| Value | Description |
|---|---|
| `"embedData"` | Default — full embedded representation |
| `"embedParent"` | Parent-facing representation |
| `"embedMinimal"` | Minimal set of fields (id + essential display fields) |

You can also define a custom getter on the model and point `embed()` at a custom key if your model defines one. For a fully custom embed shape, add the getter to the model:

```ts title="src/models/category.ts"
import { Model, Casts } from "@warlock.js/cascade";

export class Category extends Model {
  public static table = "categories";

  protected casts: Casts = {
    name: "string",
    isActive: "boolean",
  };

  public get embedIdAndName() {
    return this.only(["id", "name"]);
  }
}
```

Then reference it via `embed()`:

```ts title="src/sync/index.ts"
modelSync.register(() => {
  modelSync.sync(Category, Post, "category").embed("embedIdAndName" as any);
});
```

## 🔑 Custom Identifier Field

By default the sync matches documents by the `id` field inside the embedded object. Use `identifyBy()` to use a different field:

```ts title="src/sync/index.ts"
modelSync.register(() => {
  modelSync
    .sync(Category, Post, "category")
    .identifyBy("slug");
});
```

## 📏 Max Sync Depth

Cascade supports chained syncs — syncing a model that itself triggers another sync. The default maximum depth is `3` (the `DEFAULT_MAX_SYNC_DEPTH` constant). Override it per operation:

```ts title="src/sync/index.ts"
modelSync.register(() => {
  modelSync
    .sync(Category, Post, "category")
    .maxDepth(5);
});
```

:::info
Cascade automatically detects circular sync chains and stops them — depth limiting is an additional safeguard for long linear chains.
:::

## 👀 Watch Specific Fields

Syncing every time any field changes can be expensive. Restrict the sync to fire only when specific fields are modified:

```ts title="src/sync/index.ts"
import { modelSync } from "@warlock.js/cascade";
import { Category } from "../models/category";
import { Post } from "../models/post";

modelSync.register(() => {
  modelSync
    .sync(Category, Post, "category")
    .watchFields(["name", "slug"]);
});
```

If the category update touches only `isActive`, the sync will **not** run. This is the correct replacement for the old `updateWhenChange()` method.

## 🗑️ Behavior on Delete

When the source document is deleted you have three options.

### 1. Unset the embedded field (default)

```ts title="src/sync/index.ts"
modelSync.register(() => {
  modelSync
    .sync(Category, Post, "category")
    .unsetOnDelete();
});
```

The `category` field on matching posts is unset (removed). This is the default, so you only need to call it explicitly when you want to be clear.

### 2. Remove the target documents

```ts title="src/sync/index.ts"
modelSync.register(() => {
  modelSync
    .sync(Category, Post, "category")
    .removeOnDelete();
});
```

All posts that reference the deleted category are themselves deleted.

### 3. Leave the embedded data as-is

Simply do not call `unsetOnDelete()` or `removeOnDelete()`. Without a delete handler the embedded snapshot is preserved as it was at the time of deletion.

:::tip
Default behavior (no explicit call) leaves the embedded data in place. Call `.unsetOnDelete()` only when you want the field actively removed.
:::

## 🔧 Fluent API Reference

Every call to `modelSync.sync()` or `modelSync.syncMany()` returns a `ModelSyncOperation` instance with a full fluent API:

| Method | Description |
|---|---|
| `.embed(key)` | Set embed method: `"embedData"`, `"embedParent"`, or `"embedMinimal"` |
| `.identifyBy(field)` | Match embedded documents by this field instead of `id` |
| `.maxDepth(n)` | Maximum chained sync depth (default `3`) |
| `.watchFields([...fields])` | Only sync when one of these source fields changed |
| `.unsetOnDelete()` | Unset the target field when the source is deleted |
| `.removeOnDelete()` | Delete target documents when the source is deleted |
| `.unsubscribe()` | Manually remove this sync's event subscriptions |
| `.$cleanup()` | Alias for `unsubscribe()` (used by the HMR cleanup cycle) |
| `.getConfig()` | Return a read-only snapshot of the current sync configuration |

## 🔁 HMR-Safe Registration

In development with hot module replacement (HMR) it is critical that old event subscriptions are removed before new ones are registered. `modelSync.register()` handles this automatically:

```ts title="src/sync/index.ts"
import { modelSync } from "@warlock.js/cascade";
import { Category } from "../models/category";
import { Post } from "../models/post";
import { Comment } from "../models/comment";

const unregister = modelSync.register(() => {
  Category.sync(Post, "category").watchFields(["name"]);
  Comment.syncMany(Post, "comments");
});

// Call unregister() to tear down all syncs registered in this callback
// modelSync.register() does this automatically on re-execution during HMR
```

## 📦 TypeScript Types

The sync system exposes the following types for advanced use:

```ts
import type {
  SyncResult,
  SyncConfig,
  SyncContext,
} from "@warlock.js/cascade";
```

| Type | Description |
|---|---|
| `SyncResult` | Outcome of a sync operation — counts of affected documents and any errors |
| `SyncConfig` | Configuration object describing one sync relationship |
| `SyncContext` | Runtime context passed through a sync chain — tracks source, target, and depth |

## ⛓️ Chained Syncs and Events

All sync operations first fetch matching documents and then run `save()` or `destroy()` on each one. This means:

- A category update triggers post updates, which in turn trigger any syncs **registered on the post** (e.g. comment syncs inside posts)
- All model lifecycle events — `onSaving`, `onSaved`, `onDeleting`, `onDeleted` — fire normally on every touched document
- Cascade tracks the sync chain and enforces the `maxDepth` limit to prevent runaway chains
