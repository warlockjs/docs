# 📄 Embedded Documents

Embedded documents are a core feature of MongoDB, and they are very useful when you want to store related data inside the same document. This is a very common practice in MongoDB and is called **Embedded Relationships**.

It makes queries faster because when you fetch the post you don't have to look up the `users` collection to get the author's data — it is already stored inside the document.

Before we continue, let's create three models that we'll use throughout all of our examples.

## 🧩 Embedding Documents

There are two kinds of embedded documents:

1. Embedding a single document
2. Embedding multiple documents

### Embedding a Single Document

Think of embedding a single document as a `hasOne` relationship in SQL: the parent document contains exactly one embedded child document.

For example, a post has an `author`, so the author's data is embedded directly inside the post document:

```json
{
  "id": 1,
  "title": "Hello World",
  "content": "This is the post body",
  "author": {
    "id": 5122,
    "name": "John Doe",
    "image": "https://example.com/image.jpg"
  }
}
```

Here is how to achieve this with Cascade models:

```ts title="src/app.ts"
import { Post } from "./models/post";
import { User } from "./models/user";

async function main() {
  const author = await User.create({
    name: "John Doe",
    image: "https://example.com/image.jpg",
  });

  const post = await Post.create({
    title: "Hello World",
    content: "This is the post body",
    author: author.embedData, // embed the author's data
  });
}

main();
```

We create a new user, then create a new post and embed the author's data inside it using the `embedData` getter.

:::tip Did you know?
The `embedData` getter returns the data that will be stored when this model instance is embedded inside another document. By default it returns the full model data. **Override it** on any model to control exactly which fields get embedded.
:::

### Specifying the Embedded Data

By default, `embedData` returns the whole model document. If you only want to embed `id`, `name`, and `image` from the author, override `get embedData` on the `User` model:

```ts title="src/app/models/user.ts"
import { Model, Casts } from "@warlock.js/cascade";

export class User extends Model {
  /**
   * Collection name
   */
  public static table = "users";

  /**
   * {@inheritDoc}
   */
  protected casts: Casts = {
    name: "string",
    image: "string",
    email: "string",
    password: "string",
  };

  /**
   * Only embed these fields when this model is nested inside another document.
   */
  public get embedData() {
    return this.only(["id", "name", "image"]);
  }
}
```

Now every time `author.embedData` is used, only the following data will be stored:

```json
{
  "id": 5122,
  "name": "John Doe",
  "image": "https://example.com/image.jpg"
}
```

### Declaring Embedded Columns with `static embed`

Instead of overriding the full `get embedData` getter, you can declare a static `embed` array on the model. Cascade will read this list and pick only those columns when building the embedded payload:

```ts title="src/app/models/user.ts"
import { Model, Casts } from "@warlock.js/cascade";

export class User extends Model {
  /**
   * Collection name
   */
  public static table = "users";

  /**
   * Columns included when this model is embedded in another document.
   */
  public static embed = ["id", "name", "image"];

  /**
   * {@inheritDoc}
   */
  protected casts: Casts = {
    name: "string",
    image: "string",
    email: "string",
    password: "string",
  };
}
```

This is the recommended shorthand when you just need to whitelist a set of fields — no need to write a full getter.

:::tip
When embedding via `v.embed(Model)` in `static schema`, the `embedData` getter is used automatically to produce the nested payload, so you do not need to call it manually.
:::

## 🗂️ Embedding Multiple Documents

Embedding multiple documents means storing a list of child documents inside a single column of the parent document. For example, you can store a list of comments directly inside a post.

:::warning
It is **not recommended** to store large or unbounded lists (like comments) inside a single document. If a post accumulates many comments the document will grow very large and queries will slow down. For those cases, prefer **referencing documents** and storing comments in their own collection.
:::

### Setting Up `static embed` on the Parent

To declare which array columns hold embedded sub-documents, add a `static embed` property to the **parent** model listing those column names:

```ts title="src/app/models/post.ts"
import { Model, Casts } from "@warlock.js/cascade";

export class Post extends Model {
  /**
   * Collection name
   */
  public static table = "posts";

  /**
   * Columns that contain embedded sub-documents.
   * Cascade uses this list when serialising the model.
   */
  public static embed = ["comments"];

  /**
   * {@inheritDoc}
   */
  protected casts: Casts = {
    title: "string",
    content: "string",
    author: "object",
    comments: "array",
  };
}
```

### Pushing Embedded Documents

To add a comment into a post's `comments` array, push the comment's `embedData` directly onto the field and then save:

```ts title="src/app.ts"
import { Post } from "./models/post";
import { User } from "./models/user";
import { Comment } from "./models/comment";

async function main() {
  const author = await User.create({
    name: "John Doe",
    image: "https://example.com/image.jpg",
  });

  const comment = await Comment.create({
    content: "This is a comment",
    createdBy: author.embedData,
  });

  const post = await Post.create({
    title: "Hello World",
    content: "This is the post body",
    author: author.embedData,
    comments: [comment.embedData], // start with one embedded comment
  });
}

main();
```

### Controlling What Gets Embedded

Override `get embedData` on the child model to limit which fields are stored when it is embedded:

```ts title="src/app/models/comment.ts"
import { Model, Casts } from "@warlock.js/cascade";

export class Comment extends Model {
  /**
   * Collection name
   */
  public static table = "comments";

  /**
   * {@inheritDoc}
   */
  protected casts: Casts = {
    content: "string",
    createdBy: "object",
  };

  /**
   * Fields included when this comment is embedded inside a parent document.
   */
  public get embedData() {
    return this.only(["id", "content", "createdBy", "createdAt"]);
  }
}
```

Or use the static shorthand:

```ts title="src/app/models/comment.ts"
import { Model, Casts } from "@warlock.js/cascade";

export class Comment extends Model {
  public static table = "comments";

  /**
   * Fields included when this comment is embedded inside a parent document.
   */
  public static embed = ["id", "content", "createdBy", "createdAt"];

  protected casts: Casts = {
    content: "string",
    createdBy: "object",
  };
}
```

### Reading Embedded Data in Responses

Because `Post` declares `static embed = ["comments"]`, Cascade knows to include the embedded array when the model is serialised. You can also call `embedData` on a post instance yourself to produce the embedded payload for nesting inside yet another document:

```ts title="src/app.ts"
import { Post } from "./models/post";

async function main() {
  const post = await Post.find(1);

  // embedData returns the fields defined by static embed (or the full document
  // if embed is not declared) — ready to be stored inside another document.
  console.log(post.embedData);
}

main();
```

:::info
`embedData` is a plain getter on every model instance. It reads `static embed` if it is defined, otherwise returns the full document data. There is no method call required — just access the getter and use the result directly.
:::
