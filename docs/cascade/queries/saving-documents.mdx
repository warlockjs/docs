# 💾 Saving Documents

"Saving" covers everything from inserting a brand-new record to atomically replacing an existing one. Cascade exposes these operations as Model static methods so events, timestamps, and casting are handled for you automatically.

## ✨ Create a Single Document

```ts title="src/users/user.service.ts"
import { User } from "./user.model";

const user = await User.create({
  name: "Alice",
  email: "alice@example.com",
  isActive: true,
});
// Returns a fully-hydrated User instance
```

:::info
`create` emits `saving` and `saved` events, applies `createdAt`/`updatedAt` timestamps, and runs all casters defined on the model.
:::

## 📦 Bulk Create

Insert multiple records in parallel with `createMany`:

```ts title="src/users/user.service.ts"
const users = await User.createMany([
  { name: "Alice", email: "alice@example.com" },
  { name: "Bob",   email: "bob@example.com"   },
]);
// Returns User[]
```

:::tip
`createMany` fires create operations concurrently — order in the returned array matches the order of the input array.
:::

## 🔍 Find or Create

Return an existing record that matches the filter, or create a new one if none is found:

```ts title="src/users/user.service.ts"
const user = await User.findOrCreate(
  { email: "alice@example.com" },   // filter
  { name: "Alice", isActive: true } // data used only when creating
);
```

The second argument is merged with the filter when a new record is created, so you never have to repeat the filter fields.

## 🔄 Upsert

Update the first record matching the filter, or insert a new one if no match exists:

```ts title="src/users/user.service.ts"
const user = await User.upsert(
  { email: "alice@example.com" },   // filter / lookup
  { name: "Alice", isActive: true } // data to write
);
```

:::info
`upsert` emits `saving`/`saved` events and injects `createdAt`/`updatedAt` timestamps in both the insert and update paths.
:::

## ✏️ Update by ID

```ts title="src/users/user.service.ts"
const updated = await User.update(42, { name: "Alicia" });
// Returns the updated User instance
```

`update` is a shorthand that delegates to `updateById` internally.

## ⚛️ Atomic Update by Filter — findAndUpdate

Update **all** records matching the filter and return them:

```ts title="src/users/user.service.ts"
const archived = await User.findAndUpdate(
  { isActive: false },
  { $set: { archived: true } }
);
// Returns User[]
```

:::tip
Use MongoDB update operators (`$set`, `$inc`, `$push`, …) in the second argument for atomic field-level updates.
:::

## ⚛️ Atomic Update — findOneAndUpdate

Update the **first** matching record and return it:

```ts title="src/users/user.service.ts"
const user = await User.findOneAndUpdate(
  { email: "alice@example.com" },
  { $set: { lastLoginAt: new Date() } }
);
// Returns User | null
```

## 🔀 Replace a Document

Replace an entire document (all fields) with a new one:

```ts title="src/users/user.service.ts"
const user = await User.findAndReplace(
  { email: "alice@example.com" },
  { name: "Alice Wonderland", email: "alice@example.com", isActive: true }
);
```

:::warning
`findAndReplace` replaces the **entire** document. Any fields present in the database but absent from the replacement object will be removed. Prefer `findOneAndUpdate` with `$set` when you only want to change specific fields.
:::

## 🧙‍♂️ Pro Tips

- Use `create` for business logic paths — events, casting, and timestamps are fully applied.
- Use `upsert` when the same record might be created by concurrent processes (e.g. webhook handlers, background jobs).
- `findOrCreate` is ideal for seeding or idempotent setup scripts.
- Atomic methods (`findAndUpdate`, `findOneAndUpdate`) are safe for concurrent writes because they execute as a single MongoDB operation.
